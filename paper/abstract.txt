A hash, signature, or git diff over JSON is only meaningful once the JSON is reduced to one deterministic byte stream. Canonicalizers exist, but their correctness usually rests on a single implementation tested by its own author against its own reading of the spec -- so a misread specification passes both the code and the tests. For a primitive whose whole value is that independent parties compute the same bytes, that is a weak foundation.

We present jsoncanon, a JSON canonicalizer and toolkit built on a stronger discipline: two fully independent implementations, in Python and Nim, sharing no code and required to emit byte-identical output. A written specification is the sole arbiter, and a differential-testing gate enforces parity on every change. Because agreement alone does not prove correctness -- both could share a misreading -- the hard components are checked against external oracles: shortest-float output against a JavaScript engine, regex matching against CPython.

The canonical form normalizes numbers as decimal strings, never routing a value through a binary float, so big integers and high-precision decimals are preserved exactly and the two languages agree by construction rather than luck. This conflicts with RFC 8785, which mandates IEEE-754 rounding; we offer it as an opt-in mode that loudly reports every value whose meaning changes, and never silently loses precision.

The result is a self-contained, dependency-free package spanning the full JSON-integrity lifecycle -- lenient ingestion of real-world dialects, canonicalization, validation against four schema languages (JSON Schema, JTD, CDDL, plus GeoJSON and I-JSON profiles), deterministic binary encodings (CBOR, MessagePack), and structural transforms (JSON Pointer, Patch, Merge Patch) -- with every nontrivial component reimplemented in-tree. The novelty is the integration and the correctness contract, not any single algorithm.
